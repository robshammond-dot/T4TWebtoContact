Webform Considerations for Field Mapping

  • Name / Surname: clean two-field match to Freshsales' First Name / Last Name, no title or middle name field on the form to worry about.
  • Code + Phone Number: these are two separate fields on the form. Confirm whether Freshsales' Mobile field expects one combined international-format string (e.g. +27821234567) or separate parts, if it wants one field, the code and number need concatenating before mapping, not passed through as two raw values.
  • Prefer to be contacted by: a simple Phone/Email toggle, but worth deciding whether it just sits as a field, or actually influences which channel the follow-up sequences use, a customer who ticked "Phone" probably shouldn't only ever get email nudges.
  • Departure Point: this is a dropdown with a fixed set of options on the form. If the Freshsales field is also a picklist, its option list needs to match the webform's exactly, or values land as unmatched free text and break any reporting built on that field. Worth pulling the full list of dropdown options from the form before setting this up.
  • Start Date / End Date: confirm date format (day/month/year vs month/day/year) matches what Freshsales expects, a mismatched date format is a common, easy-to-miss import fault.
  • My dates are flexible: a simple checkbox; worth deciding whether this becomes its own Freshsales field or just informs the free-text enquiry notes.
  • Adults / Children / Childrens Ages: Adults and Children are numeric, but Childrens Ages is free text, likely comma-separated ages typed by the traveller. That's not clean, structured data, it'll need a consultant to read and interpret it manually during Consulting, not something to assume arrives report-ready.
  • Message: free text. Worth deciding whether this populates a Deal's notes field directly, or whether the consultant is expected to pull the relevant details out into structured fields themselves as part of profiling.
  • Reference and package page link: covered in the Engage and Freshsales cards above, the reference code works today, the page link is the named gap.
  • Sign up for more travel deals: covered in the Two Populations and Freshsales cards, needs to become a real subscription record, not just a submitted tick.

Two Populations: Enquiry vs Marketing Contacts

Exists today

Enquiry Contacts

Everyone currently in Freshsales asked for something, a quote, a call back, package info. Sourced from the FreshChat migration plus the webform. If someone's in Freshsales today, they're an enquiry, full stop.

Future, not built

Marketing Contacts

People who've opted into a newsletter or "more travel deals" without necessarily enquiring. None exist in Freshsales yet. Different legal basis for contact under POPIA, explicit opt-in required, not to be merged with enquiry contacts.

What WhatsApp Campaigns Actually Need

  • Explicit WhatsApp opt-in, not inferred from a generic "travel deals" tick.
  • Correctly formatted phone numbers, confirm the Code + Phone Number fields combine into what WhatsApp sending expects.
  • A defined, real subscriber list inside Freshsales to campaign against, not a manual export each time.
  • Template category compliance carried over from the existing WhatsApp/Meta classification work (Utility vs Marketing).
  • Opt-out handling consistent with the POPIA-compliant approach already built elsewhere.

Open Question: More Contacts Elsewhere?

The current Freshsales contact base comes from the FreshChat migration. Whether other contact sources exist that haven't been imported, old spreadsheets, a previous system, customer records sitting in UDMS, isn't established.
